## 🔍 核心原则

### 1. 数据优先原则

**规则**: UI 问题先查数据，不要直接改代码

**执行流程**:
```bash
# Step 1: 验证数据是否保存 (5 分钟)
python3 scripts/check_data.py

# Step 2: 如果数据正确 → UI 刷新问题
# Step 3: 如果数据错误 → 数据持久化问题
```

**触发条件**: 用户报告"UI 上看不到 X"、"X 没有显示"  
**禁止行为**: 未验证数据就直接修改代码

---

### 2. Git 历史对比法

**规则**: 复杂问题先对比 Git 历史，简单版本通常更可靠

**执行流程**:
```bash
# Step 1: 找原始实现
git log --all --oneline path/to/file.swift

# Step 2: 对比差异
git show <original_commit>:path/to/file.swift > old.swift
diff old.swift current.swift

# Step 3: 问关键问题
# - 为什么以前可以？
# - 是什么引入了问题？
# - 能恢复到简单版本吗？
```

**触发条件**: 
- 不确定当前代码是否正确
- 代码看起来很复杂
- 多次修改仍未解决

**禁止行为**: 不查看历史就盲目修改

---

### 3. SwiftUI 响应式黄金法则

**规则**: `@Published` + 计算属性 = 自动刷新，不要手动缓存

**正确模式**:
```swift
// ✅ 正确：每次都重新计算
private var filteredTodayTasks: [TaskItem] {
    switch taskFilter {
    case .all: return todayTasks  // 从 @Published 获取
    ...
    }
}
```

**错误模式**:
```swift
// ❌ 错误：破坏响应式
@State private var cachedFilteredTasks: [TaskItem] = []
private var filteredTodayTasks: [TaskItem] {
    // 条件更新缓存
    return cachedFilteredTasks
}
```

**例外情况**:
- ⚠️ 只有性能瓶颈（>1000 条）才考虑缓存
- ⚠️ 必须先测量性能，再优化

---

### 4. 一次性修改完整

**规则**: 修改前 grep 所有引用，一次性改完

**执行流程**:
```bash
# Step 1: 全局搜索所有引用
grep -r "cachedFilteredTasks" . --include="*.swift"

# Step 2: 列出所有修改点
# - TodayView.swift line 44
# - TodayView.swift line 66
# - TodayView.swift line 506
# - TodayView.swift line 529

# Step 3: 一次性全部修改
search_replace_tool(
    replacements=[
        # 所有修改点一起提交
    ]
)

# Step 4: 只编译一次
./build_tools/final_build.sh
```

**禁止行为**:
- ❌ 边编译边修改
- ❌ 分多次提交
- ❌ 遗漏引用点

## 🛠️ Bug 排查标准流程

### 流程图

```
用户报告 Bug
    ↓
第一步：检查数据文件
    ↓
数据正确吗？
├─ 否 → 检查 addTask/persist 方法
└─ 是 → Git 对比
         ↓
       找到原始实现
         ↓
       对比当前与历史差异
         ↓
       找到问题根源？
       ├─ 是 → 删除优化代码，恢复简单版本
       └─ 否 → 检查 SwiftUI 绑定/@Published
         ↓
       验证成功
```

### 时间分配

| 步骤 | 时间 | 说明 |
|------|------|------|
| 数据验证 | 5 分钟 | Python 脚本快速检查 |
| Git 对比 | 5 分钟 | 找原始实现，对比差异 |
| 最小化修改 | 10 分钟 | 删除复杂逻辑，恢复简单版本 |
| 验证 | 5 分钟 | 编译 + 重启 + 测试 |
| **总计** | **25 分钟** | 标准 Bug 排查应在 30 分钟内完成 |

## 📋 代码审查清单

### 修改前的必问问题

在修改任何代码之前，必须回答以下问题：

- [ ] **数据验证**: 我验证了数据文件吗？数据真的有问题吗？
- [ ] **Git 对比**: 我对比了 Git 历史吗？以前的简单版本是什么样的？
- [ ] **问题根源**: 我知道为什么会这样吗？还是只是在猜测？
- [ ] **全局搜索**: 我 grep 了所有引用点吗？会遗漏吗？
- [ ] **一次性改完**: 我能一次性修改所有相关点吗？
- [ ] **真的需要吗**: 我真的需要这个"优化"吗？有性能证据吗？
- [ ] **删除测试**: 如果删除这段代码，会发生什么？
- [ ] **简单优于复杂**: 有更简单的解决方案吗？

### 修改后的验证清单

- [ ] 编译成功（无 error/warning）
- [ ] 重启 App 验证
- [ ] 测试功能是否正常工作
- [ ] 检查是否有副作用
- [ ] 确认没有引入新问题

## 🎯 常见陷阱与应对策略

### 陷阱 1: "盲目修改"陷阱

**触发场景**: 
- 用户报告 Bug
- 我没有足够信息
- 开始尝试各种修改

**特征**:
- 💭 "可能是 X 问题吧..."（猜测）
- 💭 "试试这个..."（试错）
- 💭 修改后不验证效果

**正确应对**:
```
1. STOP - 停止修改
2. CHECK - 检查数据和日志
3. COMPARE - 对比历史版本
4. FIX - 基于证据修复
```

**预防咒语**: 
> "我有证据吗？没有就先收集证据，不要动手。"

---

### 陷阱 2: "过度优化"陷阱

**触发场景**:
- 看到简单代码
- 觉得"可以优化"
- 引入缓存/复杂度

**特征**:
- 💭 "这里可以加缓存提升性能"
- 💭 "这个逻辑可以优化"
- 💭 没有性能瓶颈的证据

**正确应对**:
```
1. 问：真的有性能问题吗？
2. 测：测量实际性能
3. 比：对比历史为什么简单
4. 忍：简单优于复杂
```

**预防咒语**:
> "如果以前没有优化也能工作，现在为什么需要？"

---

### 陷阱 3: "碎片化修改"陷阱

**触发场景**:
- 发现多处需要修改
- 边编译边修改
- 多次提交

**特征**:
- 编译失败 → 修改 → 再编译 → 再失败
- 循环 3+ 次

**正确应对**:
```
1. 搜：grep 所有引用点
2. 列：列出所有修改点
3. 改：一次性全部修改
4. 编：只编译一次
```

**预防咒语**:
> "修改前先 grep，一次性改完所有点。"

---

### 陷阱 4: "忽视历史智慧"陷阱

**触发场景**:
- 遇到问题
- 只看当前代码
- 不看 Git 历史

**特征**:
- 重复造轮子
- 把前人的修复当 bug
- 走前人走过的弯路

**正确应对**:
```
1. git log --oneline path/to/file
2. 找最初的简单实现
3. 问：为什么以前可以？
4. 学习并应用
```

**预防咒语**:
> "Git 历史是最好的老师，先请教历史再动手。"

## ✨ 核心洞察

> **"如果代码看起来复杂，先问自己：'以前的简单版本哪里去了？'"**

这个洞察价值连城。它可以帮助我：
- ✅ 避免过度优化
- ✅ 快速定位问题根源
- ✅ 选择正确的修复方向
- ✅ 节省大量调试时间

**具体案例**:
- TodayView 的缓存机制看起来很"智能"
- 但历史版本（commit 86d172d）根本没有缓存
- 删除缓存，立即恢复正常
- **简单就是答案**
