**修改后的 performUndo 方法**:
```swift
private func performUndo() {
    switch undoActionType {
    case .delete:
        let idsToRestore = lastDeletedTasks.map { $0.id }
        _ = taskStore.restoreTasks(deletedTaskIDs: idsToRestore)
        lastDeletedTasks.removeAll()
    // case .abandon: 分支已移除
    }
    
    closeToast()
}
```

## 🔧 技术细节

### 1. 筛选器简化

**移除前**: 6 个筛选选项
- 全部
- 待处理
- 进行中
- 已完成
- ~~已放弃~~ ← 移除
- 重要

**移除后**: 5 个筛选选项
- 全部
- 待处理
- 进行中
- 已完成
- 重要

### 2. 批量操作简化

**移除前**: 5 个操作选项
- 全选
- 删除所选
- ~~放弃所选~~ ← 移除
- 从今日任务移除 / 添加到今日任务
- 复制任务

**移除后**: 4 个操作选项
- 全选
- 删除所选
- 从今日任务移除 / 添加到今日任务
- 复制任务

### 3. 撤销类型简化

**移除前**: 2 种撤销类型
- `.delete` - 删除撤销
- ~~`.abandon`~~ - 放弃撤销 ← 移除

**移除后**: 1 种撤销类型
- `.delete` - 删除撤销

### 4. TaskStore 方法保持不变

**重要说明**: 虽然移除了 UI 层面的放弃功能，但 `TaskStore.abandonTasks()` 和 `taskStore.restoreAbandonedTasks()` 方法仍然保留在数据层，以便未来可能的功能恢复或其他调用场景。

## ⚠️ 注意事项

### 1. 数据兼容性

- **已有 abandoned 状态的任务**: 仍然会保留在数据库中，只是在 UI 层面不再显示和操作
- **筛选逻辑**: `.all` 筛选仍然会过滤掉 `status == .abandoned` 的任务

### 2. TaskStore 层保留

以下 TaskStore 方法仍然保留未删除：
- `abandonTasks(taskIDs:)`
- `restoreAbandonedTasks(taskIDs:)`
- `TaskStatus.abandoned` 枚举值

这是为了：
- 保持数据层的完整性
- 方便未来可能的功能恢复
- 避免破坏性修改导致的数据迁移问题

### 3. 后续清理建议

如果确认未来不再需要"放弃"功能，可以在后续版本中：
1. 删除 TaskStore 中的相关方法
2. 删除 `TaskStatus.abandoned` 枚举值
3. 提供数据迁移脚本，将所有 `abandoned` 状态的任务转换为其他状态
