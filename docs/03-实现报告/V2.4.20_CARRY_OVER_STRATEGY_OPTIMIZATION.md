# V2.4.20 未完成任务转入策略优化

## 📋 优化背景

### 问题描述
1. **任务冗余**：未完成任务转入今日任务后，原任务仍然在全部任务中显示，造成重复
2. **转入次数不透明**：用户不清楚任务已被转入多少次
3. **缺乏明确的状态标识**：已转入的原任务没有明确的状态标记

### 用户需求
- ✅ 任务转入今日后，原任务不应在全部任务中重复显示
- ✅ 记录每个任务的累计转入次数
- ✅ 当转入次数达到上限（默认 5 天）后，不再自动转入

## 🎯 优化方案

### 核心改动

#### 1. 转入逻辑优化 (`TaskStore+Tasks.swift`)

**修改前**：
```swift
func carryOver(taskIDs: [UUID], maxCarryDays: Int, to date: Date = Date()) {
    // ...
    let newTask = TaskItem(...)
    tasks.append(newTask)
    tasks[sourceIndex].lastCarriedOverAt = date
    tasks[sourceIndex].updatedAt = Date()
    // ❌ 原任务状态保持不变，仍然显示在全部任务中
}
```

**修改后**：
```swift
func carryOver(taskIDs: [UUID], maxCarryDays: Int, to date: Date = Date()) {
    // ...
    // 1. 创建新任务到今日
    let newTask = TaskItem(...)
    tasks.append(newTask)
    
    // 2. 将原任务标记为已放弃（避免在全部任务中重复显示）
    tasks[sourceIndex].status = .abandoned
    tasks[sourceIndex].lastCarriedOverAt = date
    tasks[sourceIndex].updatedAt = Date()
    
    print("[午夜转入] 任务 \(source.title.prefix(20)) 已转入今天 (第 \(source.carryOverCount + 1) 次)，原任务已标记为已放弃")
}
```

**关键改进**：
- ✅ 创建新任务到今日列表
- ✅ 原任务状态改为 `abandoned`（已放弃）
- ✅ 记录转入时间和次数
- ✅ 输出详细日志便于调试

#### 2. 全部任务视图筛选优化 (`AllTasksView.swift`)

**修改前**：
```swift
case .all: return allTasks.filter { $0.status != .abandoned }
case .abandoned: return allTasks.filter { $0.status == .abandoned }
```

**修改后**：
```swift
case .all: 
    // 显示所有未放弃的任务（自动转入的任务原任务会被标记为已放弃，不显示）
    return allTasks.filter { $0.status != .abandoned }
case .abandoned: 
    // 显示已放弃的任务（包括已转入的原任务）
    return allTasks.filter { $0.status == .abandoned }
```

**用户体验提升**：
- ✅ "全部任务"默认只显示有效任务（未放弃的）
- ✅ 已转入的原任务归类到"已放弃"筛选器
- ✅ 用户可以手动查看已放弃的任务（包括转入历史）

## 📊 数据模型

### TaskItem 已有字段（无需修改）

```swift
struct TaskItem: Identifiable, Codable {
    var id: UUID
    var title: String
    var status: TaskProgressStatus          // 待开始/进行中/已完成/已放弃
    var carryOverCount: Int                 // 已被转入次数
    var lastCarriedOverAt: Date?            // 最近一次被转入的日期
    var sourceTaskID: UUID?                 // 若为转入任务，记录来源任务 UUID
    // ... 其他字段
}
```

**说明**：
- `carryOverCount`: 记录任务已被转入的次数（从 0 开始）
- `lastCarriedOverAt`: 记录最近一次转入的日期
- `sourceTaskID`: 如果是转入任务，指向原任务的 UUID
- `status`: 状态变更后为 `.abandoned`

## 🔄 工作流程

### 自动转入流程（以 maxCarryDays=5 为例）

```
Day 1: 创建任务 A
  └─ carryOverCount = 0
  └─ status = .notStarted

Day 2: 午夜自动转入
  ├─ 创建新任务 A'（今日任务）
  │   └─ carryOverCount = 1
  │   └─ sourceTaskID = A.id
  └─ 原任务 A 标记为 .abandoned

Day 3: 午夜自动转入
  ├─ 创建新任务 A''（今日任务）
  │   └─ carryOverCount = 2
  │   └─ sourceTaskID = A'.id
  └─ 原任务 A' 标记为 .abandoned

...

Day 6: 午夜自动转入（最后一次）
  ├─ 创建新任务 A'''''（今日任务）
  │   └─ carryOverCount = 5
  │   └─ sourceTaskID = A'''''.id
  └─ 原任务 A'''' 标记为 .abandoned

Day 7: 不再自动转入
  └─ carryOverCount >= maxCarryDays (5 >= 5)
  └─ 跳过转入
```

### 全部任务视图展示

**场景**：用户打开"全部任务"视图

**显示内容**：
- ✅ "全部"筛选器：显示所有 `status != .abandoned` 的任务
- ✅ "已放弃"筛选器：显示所有 `status == .abandoned` 的任务（包括已转入的原任务）

**示例**：
```
全部任务 > 全部筛选器
├─ 今天的任务（新建的）
├─ 昨天的任务（未完成的）
└─ 前天的任务（未完成的）

全部任务 > 已放弃筛选器
├─ 任务 A（Day 1 创建，已转入）
├─ 任务 A'（Day 2 转入，已转入）
├─ 任务 A''（Day 3 转入，已转入）
├─ 任务 A'''（Day 4 转入，已转入）
└─ 任务 A''''（Day 5 转入，已转入）
```

## 🎯 优势总结

### 技术优势
1. ✅ **数据一致性**：原任务和新任务都有明确的状态标识
2. ✅ **可追溯性**：通过 `sourceTaskID` 可以追踪完整的转入链路
3. ✅ **性能优化**：不需要额外的过滤逻辑，利用现有状态字段

### 用户体验优势
1. ✅ **清晰的视觉反馈**：全部任务中看不到重复任务
2. ✅ **透明的转入历史**：可以在"已放弃"中查看转入记录
3. ✅ **智能的限制机制**：达到上限后自动停止转入
